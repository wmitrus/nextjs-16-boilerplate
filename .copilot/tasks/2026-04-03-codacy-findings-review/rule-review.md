# Rule Review

## Objective

- Review the repeated Codacy rules against repository ownership, runtime priority, and existing triage evidence.
- Decide whether each rule should be kept, narrowed, documented for AI, or demoted for repository or path scope.
- Separate production-runtime value from test, script, and dev-only noise without changing code or security docs in this step.

## Docs vs Code Drift

- There is no material code-vs-doc architecture drift in the reviewed findings. The drift is between Codacy's default rule behavior and the repository's actual ownership model.
- Existing security guidance already supports two repeated false-positive themes: explicit dispatch maps in `SEC-04` and confined static-path access in `SEC-05` / `SEC-12`.
- Current guidance does not yet explicitly capture two repository patterns repeatedly hit by Codacy: own-key enumeration into fresh or null-prototype accumulators, and the requirement that reusable fs helpers enforce confinement at the point of access rather than relying on callers.

## Architectural Assessment

- The strongest repository-level conclusion is that rule value depends more on layer ownership than on raw finding count. Rules that still detect trust-boundary mistakes in `src/app`, `src/core`, `src/modules`, `src/security`, and runtime-supporting `src/shared/lib` should remain active even when most hits are false positives elsewhere.
- Tests, E2E helpers, local scripts, and editor tooling are currently overrepresented in the finding volume. Leaving those paths unscoped causes the main review queue to optimize for scanner noise instead of production runtime risk.
- `security/detect-object-injection` should remain active for bootstrap, route, auth, and server-runtime code because it still exposes a live untrusted-input pattern. The same rule is low-value for test scaffolding, sanitizer accumulators, and finite-key environment alias lookups.
- `security/detect-non-literal-fs-filename` should remain active where a reusable helper or operator-facing script can become a path-traversal boundary. It is low-signal for repository-owned env file readers and confined artifact paths.
- The non-security rules split cleanly by ownership: `@typescript-eslint/no-explicit-any` still adds value in core runtime infrastructure, `@typescript-eslint/no-deprecated` adds value in production code but not in intentional compatibility tests, and `prettier/prettier` adds little value in Codacy because the repository already enforces formatting elsewhere.
- AI instruction updates are justified only where the scanner repeatedly collides with established repository-safe patterns or where a reusable hardening rule should be made explicit. That threshold is met for `security/detect-object-injection` and `security/detect-non-literal-fs-filename`, not for the other rules reviewed here.

## Risks

- Broadly disabling `security/detect-object-injection` or `security/detect-non-literal-fs-filename` would hide the two runtime-relevant surfaces still worth architectural scrutiny: bootstrap query-param lookup and reusable logger path creation.
- Leaving the current path mix unchanged will keep future review cycles dominated by tests, E2E, scripts, and local tooling, reducing signal on runtime ownership boundaries.
- Updating AI instructions without keeping the scope narrow would risk over-normalizing patterns that are only safe under specific constraints such as finite unions, own-key enumeration, or repository-confined paths.

## Recommended Next Action

- Carry these rule decisions into the remediation step without changing code yet: preserve runtime scrutiny for `security/detect-object-injection` and `security/detect-non-literal-fs-filename`, demote the clearly low-signal test and dev-only rules, and treat `prettier/prettier` as out of scope for security review.
- Queue a later Security & Auth documentation pass to clarify the two AI-guidance gaps identified here, but do not edit `docs/ai/general/SECURITY_CODING_PATTERNS.md` in this step.
