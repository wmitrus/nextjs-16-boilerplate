## Runtime Analysis

### Server vs Client Placement

`getNrBrowserCdnConfig()` is called from `RootLayout` — a **synchronous Server Component**. The function reads only from `env.*` (T3-Env) which wraps `process.env`. These are server-side env vars — correct placement. No client-bundle exposure.

The NREUM configuration (including `licenseKey`) is rendered into inline HTML via `dangerouslySetInnerHTML`. The browser license key is intentionally public (NR Browser license keys are designed to be visible in HTML). No security concern here.

### Prerendering Behavior (`cacheComponents: true`)

With `cacheComponents: true` (PPR model), the `RootLayout` shell is eligible for static prerendering. The `env.*` reads in the synchronous layout function are evaluated:

- **Local dev (`pnpm dev`)**: Per-request — env vars are fresh from `.env.local`
- **Vercel build**: At build time — env vars from Vercel project settings

**Critical**: If `NEW_RELIC_BROWSER_ENABLED=true` is NOT set in Vercel project settings at build time, `getNrBrowserCdnConfig()` returns `null` → prerendered HTML has no `<script>` tags → all Vercel requests served without NR scripts, even if vars are added later without a redeploy.

**A redeploy is required** every time browser monitoring env vars change on Vercel.

### Script Strategy Analysis

The layout uses `strategy="beforeInteractive"` for both the inline config script and the CDN agent script. This is the **correct and only supported strategy** for a monitoring agent that must:

- Record page load timing (LCP, FCP, TTFB)
- Capture XHR/Fetch from page bootstrap
- Catch JavaScript errors during hydration

`beforeInteractive` in `app/layout.tsx` (root layout) is fully supported by Next.js 16 App Router. The constraint applies only to page segments, not the root layout.

### Route Handler (`/observability/new-relic-browser.js`)

The route handler currently serves only the APM-linked path:

- Gates on `env.NEW_RELIC_ENABLED && env.NEW_RELIC_LICENSE_KEY`
- Returns empty when APM is not connected (Vercel — expected)
- CDN branch was correctly removed in the previous fix (chat `80f77b48`)

The APM-linked path is retained for local dev where `NEW_RELIC_ENABLED=true` and the NR Node.js agent connects. Layout uses `afterInteractive` for this path (APM mode), correctly falling back to `afterInteractive` for local-only telemetry.

### `connection()` Requirement

Per AGENTS.md, `connection()` is required when `getAppContainer()` is called (to avoid Pino timestamp prerender errors). `RootLayout` does NOT call `getAppContainer()` — it reads only `env.*`. The `connection()` requirement does NOT apply here.

### Caching / Revalidation

The `<head>` content generated by `RootLayout` (including NR scripts) is part of the prerendered shell. No per-request revalidation is needed — the beacon URL and keys are constants for a given deployment. Correct behavior.
